# just-ai architecture decisions

## Companion and module boundaries
The upstream `just` package remains behaviorally unchanged and has no dependency on just-ai. `just-ai` communicates through the public JSON dump and direct argv-based process execution. Physical modules separate CLI, inspection, AI contracts, proposals, providers, application, and domain. The Tauri/React GUI and JSON-RPC/stdio MCP server are independent workspace adapters that depend inward on core.

## AI safety, transport, and context
AI output is an untrusted typed proposal, never an executable action. Local deterministic code validates syntax, dependencies, risk, workspace paths, and reviewed content before atomic application. Providers implement `AiProvider`. OpenAI uses Responses with strict Structured Outputs; the default is `gpt-5.6-terra` with `reasoning.effort=none`. Ollama uses native `/api/chat` with schema format, non-streaming output, and temperature zero. Generic OpenAI-compatible servers retain Chat Completions. Returned text is validated locally against the operation schema. Project context is allowlist-only, bounded, excludes dotenv files, and redacts likely credentials.

## Execution and history
Execution is two-phase: prepare obtains preview, risk, and policy; execute re-prepares before confirmation. Frontends submit recipe names and argv arrays, never shell strings. Every just invocation inserts `--` between just-ai-controlled options and the recipe name/arguments, preventing dash-prefixed recipe values from being reinterpreted as just options. Before dry-run, core reads the non-evaluating `just --dump --dump-format json` representation through a shared private adapter and rejects every structured function-call expression and every non-empty `dotenv_command` setting. This conservative preflight prevents project-defined evaluation before authorization; execute and streaming execute repeat it when they re-prepare at the authorization boundary. Dump and dry-run subprocesses share a private concurrent capture adapter that limits stdout and stderr to 8 MiB each, kills the direct child on overflow, and never exposes a caller-controlled limit. Streaming execution owns a platform-specific process-tree supervisor. Unix uses an isolated process group; Windows uses a private Job Object configured with `JOB_OBJECT_LIMIT_KILL_ON_JOB_CLOSE`, assigns the direct child, and calls `TerminateJobObject` on cancellation. Assignment failure kills and waits for the child before returning an error. History is bounded, atomic, per-project JSONL with redacted 16 KiB tails. Records include argv and cancellation state with serde defaults for older JSONL. Canonical project roots unify CLI and GUI history.

## Desktop contract
The desktop adapter renders `ContextParameter` and never parses justfiles. Singular values remain one argv element; plus/star values use one element per line. History renders timestamps, argv, cancellation state, and expandable redacted tails through bounded Tauri IPC.

## MCP contract
`apps/just-ai-mcp` is a separate newline-delimited JSON-RPC/stdio adapter. Stdout is protocol-only. `lib.rs` owns only newline-delimited stdio transport; `protocol.rs` owns JSON-RPC parsing, version negotiation, and dispatch; `catalog.rs` owns the compile-time prompt/resource allowlists; `tools.rs` is the only MCP module depending on just-ai core and exposes read-only inspection/dry-run operations. MCP tool schemas accept neither an executable path nor a project root: production resolves the literal `just` command from the server-controlled process environment and uses the process working directory, with alternate binary/root injection private to the adapter's internal test seam. One adapter process is scoped to one project. Runtime per-tool argument allowlists reject non-object arguments, unknown keys, and client-controlled boundary fields before any filesystem or process operation; these failures are returned as MCP tool results with `isError: true`. Input frames are incrementally read with a 1 MiB bound; oversized frames are discarded through their terminating newline and non-UTF-8 frames are parsed as errors, after which service continues. The server supports published MCP versions 2024-11-05, 2025-03-26, 2025-06-18, and 2025-11-25: known client versions are echoed, unknown versions receive the latest supported version. Envelopes are validated before notification classification: malformed input uses standard JSON-RPC parse/invalid-request errors and a safe null identifier when necessary, while valid notifications produce no response. The server exposes only read-only `inspect_project`, `doctor`, and `prepare_run`; preparation delegates to the core preflight and guarded dry-run. There is deliberately no execution or write tool. Tool results include both `structuredContent` and serialized text content, and annotations declare read-only/non-destructive behavior. The server exposes argument-free `implement`, `review-architecture`, `refresh-index`, `system`, and `verify` prompts compiled directly from canonical `agent/` files. Prompt lookup errors use JSON-RPC invalid params, and prompt messages use the protocol-supported `user` role. Architecture, roadmap, and accepted ADRs are compile-time `just-ai://docs/*` resources. The fixed allowlist exposes no path templates, arbitrary filesystem reads, subscriptions, or change notifications; unknown URIs return MCP resource-not-found.

## Compatibility testing
Versioned JSON dump fixtures cover basic, rich, and Windows `just 1.54.0` values. The Windows fixture preserves drive-letter source/default paths, PowerShell/cmd bodies, structured interpolation, and nested modules on every host; Windows CI additionally asserts native absolute-path semantics. Provider mocks assert transport-specific request and response contracts. Execution tests assert the exact prepare/execute argv including the `--` option boundary. Structural preflight tests distinguish function-call arrays from literal function-like text and reject non-empty `dotenv_command`; marker-based process tests prove both classes are stopped before dry-run begins. Bounded-capture tests cover the exact byte boundary and overflow on both stdout and stderr. History tests cover migration defaults and storage invariants. Unix and Windows cancellation tests exercise descendant termination and inherited output-pipe closure; a dedicated Windows CI job compiles, lints, and runs the native Job Object path. MCP unit tests cover bounded frame recovery, protocol negotiation, invalid requests, notification silence, strict tool inputs, read-only discovery, actual guarded dry-run preparation, canonical prompts, and allowlisted resources. Black-box integration tests spawn the MCP binary and verify newline framing, protocol-only stdout, silent notifications, response ordering, and recovery after malformed JSON. Stderr remains available for diagnostics. Layered CI verifies core, desktop, and MCP workspaces.

## Agent workflow
Code discovery and impact analysis use Codebase Memory MCP first. Structural changes require tests, formatting, Clippy, documentation/ADR updates, re-indexing, and graph verification. `agent/commands/verify.md` is the canonical layered verification playbook shared verbatim by `just-ai agent verify` and the MCP `verify` prompt.
